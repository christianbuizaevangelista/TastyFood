# Tasty Food Manufacturing Inc. ‚Äî Distribution Management System

A multi-tier distribution management web application for **Tasty Food Manufacturing Inc.**
(consumer brand: **Juan Palaman**, General Trias, Cavite). It models a 4-tier
distribution hierarchy with role-based access, inventory, purchase orders, POS,
sales reporting, approvals, and KPI tracking.

> **Branding note:** the reference page `facebook.com/juanpalamanofficial` is login-gated,
> so the visual identity here uses a warm Filipino-food palette (tomato red + golden
> accent + cream). All brand colors live in [`frontend/tailwind.config.js`](frontend/tailwind.config.js)
> and the logo mark in the sidebar/login ‚Äî swap them in one place to match the real guide.

## Distribution model

```
PRINCIPAL  (Tasty Food Mfg. Inc.)        discount: 0%   (sets SRP)
  ‚îî‚îÄ‚îÄ PROVINCIAL DISTRIBUTOR              buys at 20% off SRP
        ‚îî‚îÄ‚îÄ CITY DISTRIBUTOR             buys at 15% off SRP
              ‚îî‚îÄ‚îÄ RESELLER               buys at 8% off SRP
```

- **Each tier buys only from the level directly above it.**
- **Buyer price = SRP √ó (1 ‚àí tier discount).**
- **Trade** orders/sales deduct the seller's inventory. **Drop Ship** orders/sales do **not**.
- **Onboarding/approvals:** Principal approves Provincial & City distributors; Provincial
  approves Resellers. New accounts start **PENDING** and cannot transact until **APPROVED**.
- **Scoping:** every query is restricted to the requester's org **and its descendants**.

## Quick start (Docker ‚Äî recommended)

Requires Docker Desktop.

```bash
docker compose up --build
```

This starts Postgres, runs `prisma db push` to create the schema, seeds demo data,
and launches both services:

- **Frontend:** http://localhost:5173
- **API:** http://localhost:4000/api  (health: `/api/health`)

The database is seeded only when empty, so restarts keep your data.

---

## Local development (without Docker)

You need Node 18+ and a local PostgreSQL.

### 1. Backend

```bash
cd backend
cp .env.example .env          # edit DATABASE_URL / JWT_SECRET as needed
npm install
npx prisma generate
npx prisma migrate dev --name init   # creates schema + migration files
npm run db:seed                       # load demo data
npm run dev                           # http://localhost:4000
```

> No DB yet? Use `npx prisma db push` instead of `migrate dev` to create the schema
> without migration files. Migration files are generated by `migrate dev` once a
> database is reachable.

### 2. Frontend

```bash
cd frontend
cp .env.example .env
npm install
npm run dev                  # http://localhost:5173 (proxies /api to :4000)
```

## Modules

| Module | Description |
|--------|-------------|
| **Dashboard** | Role-aware KPI cards (sales, inventory value, pending approvals, active members, low stock) + revenue trend, trade-vs-dropship split, top performers. |
| **Inventory** | Per-org stock, low-stock alerts, per-SKU append-only stock ledger, manual adjustments. |
| **Purchase Orders** | `DRAFT ‚Üí SUBMITTED ‚Üí APPROVED ‚Üí FULFILLED ‚Üí RECEIVED` (+ `CANCELLED`). Buyer's tier discount applied automatically. Trade fulfill decreases seller stock; receive increases buyer stock. Drop-ship changes neither. |
| **POS** | Direct sales (Trade/Drop Ship), auto-priced with discount, printable receipt. |
| **Sales Report** | Filter by date, tier, distributor, product, distribution type & channel. Downstream rollup. CSV export. |
| **KPI & Leaderboards** | Per-org revenue vs target, growth %, active members, PO fulfillment rate, inventory turnover; ranked top/bottom. |
| **CRM / Accounts** | Manage downstream accounts, contacts, targets, order history. Onboard new accounts. |
| **Approvals** | Org onboarding + PO approvals awaiting the requester's decision. |
| **Products** | Principal-only master catalog (SRP drives all pricing). |
| **Membership Activation** | Activate/deactivate downstream accounts (in CRM). |

## Key business rules (enforced in the backend)

1. Buyer price = `SRP √ó (1 ‚àí tier discount)` (see [`backend/src/lib/pricing.ts`](backend/src/lib/pricing.ts)).
2. POs flow only to the immediate parent tier.
3. Trade adjusts inventory; Drop Ship does not.
4. Unapproved/inactive orgs cannot create POs or record sales.
5. Every query is scoped to the requester's org + descendants
   (see [`backend/src/lib/scope.ts`](backend/src/lib/scope.ts) and
   [`backend/src/middleware/auth.ts`](backend/src/middleware/auth.ts)).

|--------|-------|-------|

| GET | `/inventory` ¬∑ `/inventory/alerts` ¬∑ `/inventory/ledger` ¬∑ POST `/inventory/adjust` | |
| GET/POST | `/products` (POST = Principal) | |
| GET/POST | `/purchase-orders` + `/:id/{submit,approve,reject,fulfill,receive,cancel}` | |

| GET | `/sales` ¬∑ `/sales/export.csv` | filters: from,to,tier,sellerOrgId,productId,distributionType,channel |

| GET/POST | `/approvals` ¬∑ `/approvals/:id/decide` | org onboarding |

### Data models
`Organization` (type, parentId, discountRate, status, isActive) ¬∑ `User` (role) ¬∑
`Product` (sku, srp) ¬∑ `Inventory` ¬∑ `StockLedger` ¬∑ `PurchaseOrder` + `PurchaseOrderItem`
(distributionType, status) ¬∑ `Sale` + `SaleItem` (distributionType, channel) ¬∑
`Approval` (ORG_ONBOARDING / PO_APPROVAL) ¬∑ `KPIRecord`.
